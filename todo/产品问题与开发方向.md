## 一、备份流程问题

### 1.1 备份目标路径「虚设」

- `home_tab.py` 的 `target_path_label` 初始值来自 `self.storage.base_dir`，但 `self.storage` 在 `HomeTab.__init__` 时就固定为 `LocalStorage` 实例。用户浏览路径存入 label，`_backup()` 执行时才动态创建新 storage——浏览行为不验证路径可用性
- ZIP 模式下，`manifest_base_dir` 和 `ZipStorage` 用了同一路径 → manifest 和 zip 文件混在一起

**解法**：browse 后检查 `os.access(path, W_OK)`；ZIP 模式 manifest 路径与输出路径分离。

### 1.2 文件收集可能产生路径冲突

`file_utils.py` 中 `collect_files()` 递归目录时：

```python
rel = str(f.relative_to(expanded.parent))
```

同一规则下不同文件如果 `expanded.name` 相同（跨子目录同名文件），后收集的会覆盖前者。不过当前按文件名做 key 的实际影响有限。

**修复**：改为 `expanded` 到文件的全相对路径。

### 1.3 进度条粗糙

备份/恢复进度只有离散跳转（每个配置完成跳一次），没有基于总文件数或总字节数的精确百分比。

**改进**：信号改为 `(current, total)`，UI 计算百分比。

### 1.4 定时任务脚本路径硬编码

`settings_tab.py:103`：

```python
script = Path(__file__).resolve().parent.parent.parent / "main.py"
```

打包为 exe 后此路径不存在。定时任务本质靠 `schtasks.exe` 调用 `python main.py --silent-backup`，打包后应改为 exe 自身路径。

## 二、恢复流程问题

### 2.1 恢复不能选目标路径

`_restore_single()` 调用 `RestoreWorker(cfg, self.storage, backup_id, None, ...)`，`restore_dir=None`。用户只能恢复到匹配到的原始路径，**没有 UI 入口选择自定义恢复位置**。

**改进**：home_tab 右侧新增「恢复目标」路径选择器（同备份侧），传入 RestoreWorker。

### 2.2 批量恢复中断即终止

`_restore_error()` 直接 `_restore_session_queue = []` 清空剩余队列，没有「跳过失败继续下一个」的选项。已恢复的文件不回滚，处于部分恢复的不一致状态。

**改进**：改为记录失败 → 跳过 → 继续下一个 → 最终汇总「N 成功 / M 失败 / K 跳过」。

### 2.3 恢复前备份（undo）可能覆盖文件

`restore_engine.py:115`：

```python
shutil.copy2(dst, backup_dir / dst.name)
```

不同目录下同名文件的备份会互相覆盖，每个文件名只能保留最后一次。

**修复**：`backup_dir / dst.relative_to(dst.anchor)` 保持完整目录结构。

### 2.4 恢复结果没有文件级展示

`RestoreResult` 只返回了 `files_restored` 列表，GUI 只弹窗显示 `result.config_name`。用户看不到具体恢复了哪些文件、哪些失败、哪些跳过。

**改进**：扩展 `RestoreResult`，增加 `files_failed: list[tuple[Path, str]]`；UI 展示详细列表。

### 2.5 两个存储后端的 restore 行为不一致

- `LocalStorage.restore()` 直接 `shutil.copy2` 版本目录下所有文件，不走 `restore_engine.py` 的 `_resolve_target_paths` 路径解析
- `ZipStorage.restore()` 先解压再复制

如果外部代码直接调 `storage.restore()`（未来 CLI 扩展），会绕过路径安全性校验。当前 GUI 用的是 `RestoreWorker`（走 `storage.get_files()` + `_resolve_target_paths` + `shutil.copy2`），不直接调 `storage.restore()`。

**改进**：统一入口，`storage.restore()` 只做文件取出，路径解析全部由 `RestoreWorker` 处理，或让 `storage.restore()` 内部也调用 `_resolve_target_paths`。

## 三、存储层问题

### 3.1 Manifest 位置跟随备份目录

```python
# backup_engine.py:378
manifest_base_dir = Path(target_path_str)
```

用户切换备份目标目录后 manifest 丢失 → 增量备份退化为全量备份。

**改进**：manifest 统一存放在固定位置（如 `%LOCALAPPDATA%/WinConfigBKP/manifests/` 或 `get_app_root() / ".manifests"`），与备份输出目录解耦。

### 3.2 ZipStorage 不支持增量追加

每次 `save()` 输出新的 `.zip` 文件，无法像 LocalStorage 那样按文件粒度增量。大文件配置在 ZIP 模式下每次都是全量开销。

**改进**：ZipStorage 支持分卷或追加到已有 zip（需要 `zipfile.ZipFile` append 模式限制较多，可能需要记录文件级 diff 再打包变更集）。

### 3.3 LocalStorage.restore 无 source_type 路径解析

直接复制 `ver_dir` 下所有文件，对于 `data_paths` 配置无法正确路由到 data 目标路径（虽然有 `source_type` 记录在 metadata 中）。

## 四、扫描问题

### 4.1 Scanner 模糊匹配过于简陋

`scanner.py:52-55`：

```python
rule_keywords = name.lower().split()
if any(kw in entries for kw in rule_keywords):
```

规则名按空格分词，去扫描目录的子目录名集合里做 `in` 匹配。「Windows Terminal」分词成 `["windows", "terminal"]`，会在 `%APPDATA%` 下找叫 `windows` 或 `terminal` 的目录——几乎不可能命中。

**改进**：为每条规则加 `detect_keywords` / `detect_paths` 字段，或直接检查规则中定义的路径本体（paths 里的路径存在就匹配）。

### 4.2 Scanner 只检查 paths 不检查 data_paths

扫描只遍历 `rule.get("paths", [])`，不包含 `data_paths`。某些程序只有 data 路径没有 config 路径时不会被扫描到。

**改进**：统一注释、README、UI 文本，保持一致。
